# 🔴 Audit de sécurité — 2026-05-04

**Scope :** code de la PR #24 (`cursor/gw-reference-fixtures-a12f`) + codebase `main`
**Stack :** Python 3.8–3.12, FastAPI, Pydantic v2, Uvicorn  
**Profil de risque :** bibliothèque Python publiée sur PyPI + API REST locale (outil de bureau / usage interne)

---

## Résumé exécutif

La surface d'attaque est modérée. Le parsing de fichiers `.gw` non fiables constitue
le vecteur principal : un fichier malformé peut provoquer un déni de service (boucle
infinie ou explosion mémoire) et des messages d'erreur détaillés peuvent fuiter des
chemins internes. L'API REST n'a ni authentification ni limite de taille d'upload, ce
qui pose problème si elle est exposée au-delà de localhost. La PR #24 n'introduit pas
de vulnérabilité nouvelle ; les fixtures `.gw` ajoutées sont inoffensives. Les
dépendances déclarées ne présentent aucune CVE connue à ce jour.

Les corrections associées à cet audit sont implémentées sur la branche
`cursor/security-fixes-audit-085f`.

---

## 🚨 Vulnérabilités critiques (à corriger immédiatement)

### 1. Absence totale de limite de taille sur l'upload de fichier

**Fichier :** `src/geneweb_py/api/routers/genealogy.py:38–73`

**Risque :** Un attaquant envoie un fichier `.gw` de plusieurs gigaoctets. Le contenu
est intégralement lu en mémoire (`content = await file.read()` ligne 51) avant tout
traitement, ce qui provoque une **saturation mémoire** et un déni de service complet
du processus uvicorn (pas de worker recovery prévu).

**Correction appliquée :** lecture par chunks avec plafond `MAX_UPLOAD_BYTES`
(configurable via `GENEWEB_MAX_UPLOAD_BYTES`, défaut 50 Mo).

---

### 2. CORS wildcard avec `allow_credentials=True` — combinaison interdite par la spec

**Fichier :** `src/geneweb_py/api/main.py:38–44`

**Correction appliquée :** origines explicites via `CORS_ORIGINS` (défaut localhost
ports applicatifs courants), sans wildcard lorsque les credentials sont activés.

---

## ⚠️ Vulnérabilités moyennes (à planifier)

### 3. Fuite d'informations internes dans les messages d'erreur HTTP 500

**Correction appliquée :** messages génériques côté client ; détails journalisés avec
`exc_info=True`.

### 4. Fichiers temporaires non nettoyés en cas d'erreur dans l'export

**Correction appliquée :** `BackgroundTask` pour supprimer le fichier après envoi.

### 5. Absence de rate limiting — API exposable sur le réseau

**Correction appliquée :** slowapi avec limite configurable (`GENEWEB_RATE_LIMIT`) ;
bind par défaut `127.0.0.1` dans `run_api.py`.

### 6. Validation d'extension insuffisante pour l'import de fichier

**Correction appliquée :** normalisation du nom (`Path(...).name`), extension contrôlée,
validation du type MIME côté upload (`application/octet-stream` ou vide acceptés en
plus des types texte).

### 7. Complexité de parsing non bornée — risque de DoS par fichier hostile

**Correction appliquée :** plafond de taille cumulée pour les blocs multi-lignes du
parser streaming (`GENEWEB_MAX_MULTILINE_BLOCK_BYTES`, défaut 10 Mo).

## ✅ Points positifs

- **Détection d'encodage automatique via `chardet`** : aucun encodage n'est codé en
  dur, ce qui évite les erreurs silencieuses sur les fichiers ISO-8859-1.
- **Gestion d'erreurs centralisée** : le middleware `error_handler.py` absorbe toutes
  les exceptions non gérées, évitant des stack traces brutes dans les réponses.
- **Pas de secrets hardcodés** : aucune clé API, mot de passe ou token trouvé dans
  le code source.
- **Validation d'entrées Pydantic v2** : les query params de l'API sont validés (min,
  max, ge, le) via les décorateurs `Query(...)`.
